# Releasing

Stable plugin updates are delivered through a packaged GitHub release asset zip, not GitHub's auto-generated source archive. This keeps repository-only folders like `offline-app-builder/` off live WordPress sites.

## Rules

- Keep the `Version:` header in `language-learner-tools.php` aligned with the release tag.
- If the release bumps the plugin version, start the release commit subject with the new version number, for example `5.8.1 - Package stable updates from GitHub release assets`.
- Upload the packaged plugin zip generated by `scripts/build-release-package.sh`.
- Do not rely on GitHub's default `Source code (zip)` or `Source code (tar.gz)` assets for stable releases.
- `Main` is the production channel. `Dev` still pulls the GitHub `dev` branch directly and should stay testing-only.

## Release Steps

1. Finish the release changes and update the plugin `Version:` header.
2. Commit the release changes.
3. Create and push a tag that matches the version, for example `v5.8.0`.
4. Build the release asset from that tag:

   ```bash
   ./scripts/build-release-package.sh v5.8.0
   ```

5. Create a GitHub release for the same tag.
6. Upload the generated zip from `dist/`, for example `dist/language-learner-tools-5.8.0.zip`.
7. Publish the release.

## What The Script Does

- Builds the plugin zip with `git archive`.
- Uses `.gitattributes` `export-ignore` rules to exclude repository-only files and folders.
- Forces the zip root directory to `language-learner-tools/`, which matches the installed plugin directory.

## Verification Checklist

- The uploaded asset name starts with `language-learner-tools` and ends with `.zip`.
- The zip root folder is `language-learner-tools/`.
- The zip does not contain `offline-app-builder/`, `tests/`, or other export-ignored paths.

If a stable GitHub release is published without a matching plugin zip asset, `Main` channel sites will not receive an update. That failure mode is intentional so production installs never fall back to shipping the raw repository archive.
